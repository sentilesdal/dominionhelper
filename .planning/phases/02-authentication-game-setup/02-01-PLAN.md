     - Depends on `gamePage`
     - Calls `validateCredentials()` — if null, calls `test.skip(true, 'DOMINION_USER and DOMINION_PASS required in .env')` and returns
     - Waits for Angular to bootstrap: `await gamePage.waitForFunction(() => !!(window as any).angular, { timeout: 15000 })`
     - Checks if already logged in: `await gamePage.evaluate(() => !!document.querySelector('.lobby-page'))` — if true, skip login
     - Uses Playwright's `page.fill()` for the login form (fills username and password fields)
     - Intercepts login API response: `const responsePromise = gamePage.waitForResponse(resp => resp.url().includes('login') || resp.url().includes('auth'), { timeout: 30000 })`
     - Clicks login button via role-based locator: `await gamePage.getByRole('button', { name: /log\s*in|sign\s*in/i }).click()`
     - Verifies response status is 200; throws if not
     - Has retry logic: wraps login attempt in a for loop (2 attempts max). On first failure, calls `await gamePage.reload()` then retries. On second failure, captures screenshot to `test-results/login-failure.png` and throws `Error('Login failed after retry: ...')`
     - On success, calls `await use(gamePage)` (the authenticated page IS the gamePage after login)
   - NOTE: Exact login form selectors are UNKNOWN at plan time. The executor MUST use `--headed` mode (`npx playwright test e2e/auth.spec.ts --headed`) to discover the actual form selectors on dominion.games. Use Playwright's resilient locators (role-based, text-based, input type) wherever possible. The research suggests `input[type="text"]` for username and `input[type="password"]` for password, with a button matching `/log\s*in|sign\s*in/i`, but these MUST be verified visually.
